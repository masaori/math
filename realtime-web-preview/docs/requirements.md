# 構造化テキストのリアルタイム Web プレビュー — 要件定義

## 1. 目的（やりたいこと）

ローカル（Mac）で編集している「tex 入り構造化テキスト」の**レンダー結果**を、
**別 PC やスマホのブラウザからリアルタイムに閲覧**できるようにする。

- このアプリは**特定の研究ドメインに依存しない汎用ツール**とする。
  リポジトリ内の各プロジェクト（例: `exact-solution-of-2d-ising-model/structured-latex/`）が
  生成する構造化テキストを**入力ソースの一つ**として受け取る、という位置づけ。
- 編集はローカルで行う（エディタは問わない）。
- 編集内容が保存されると、閲覧側のブラウザ表示が**自動で更新**される（手動リロード不要）。
- 数式は KaTeX でレンダリングする。

## 2. 用語と前提

- **構造化テキスト**: ブロック列（`ConvertedBlock[]`）として表現されたデータ。
  - 数式は LaTeX 文字列（`math` / `displayMath` ノード）として保持される。
  - 各ブロックは `kind`（theorem/definition/claim/remark/note）, `title`, `statement`,
    `proof`, `notes`, `sourcePath`, `sourceOrdinal` 等を持つ。
  - 現時点の具体例 / リファレンスとなる入力ソースは
    `exact-solution-of-2d-ising-model/structured-latex/`（スキーマは同ディレクトリの
    `schema.d.ts`、データは `content/*.mjs`、検証は `tools/validate-content.mjs`）。
  - ただし本アプリはこのプロジェクト固有の事情に**依存しない**。スキーマ（入力契約）を
    満たす任意のソースを受け付けられる設計とする。
- **レンダー結果**: 上記ブロック列を HTML 化し、数式を KaTeX で描画したもの。

## 3. スコープ

### 3.1 In scope（今回やる）
- 構造化テキストを HTML + KaTeX で表示する**ビューア**。
- ファイル変更を検知して閲覧ブラウザを**自動更新**する**ライブリロード**。
- **同一ネットワーク内**の別 PC / スマホからの**閲覧**（read-only）。
- 入力ソースを**設定で差し替え可能**にする（特定プロジェクトに固定しない）。

### 3.2 Out of scope（今回やらない）
- ブラウザ側からの**編集**・双方向同期（閲覧専用でよい）。
- 複数人での同時編集・コンフリクト解決。
- 認証・アクセス制御の作り込み（§7 でゆるい前提を置く）。
- 構造化テキスト自体の生成・変換（各プロジェクト側の責務）。
- Typst / PDF 出力の置き換え（別軸の作業）。

## 4. 利用シナリオ

1. ローカルで構造化テキスト（例: `structured-latex/content/*.mjs`）を編集して保存する。
2. ローカルでプレビュー用プロセス（ローカルサーバ）を 1 つ起動しておく。
3. 別 PC やスマホのブラウザでローカル機の URL（例: `http://<LAN IP>:<port>/`）を開く。
4. 保存するたびに、別 PC / スマホの表示が**数秒以内に自動更新**される。

## 5. 機能要件

| ID | 要件 | 優先度 |
|----|------|--------|
| F-1 | 設定された入力ソースのブロック列を読み込み、HTML として表示する | 必須 |
| F-2 | `math` / `displayMath` を KaTeX で描画する | 必須 |
| F-3 | `kind` ごとに見出し・体裁を分けて表示（theorem/definition/claim/remark/note） | 必須 |
| F-4 | `title`, `statement`, `proof`, `notes` を構造的に表示する | 必須 |
| F-5 | ファイル保存を検知し、閲覧ブラウザを自動更新する（ライブリロード） | 必須 |
| F-6 | LAN 内の別端末（PC / スマホ）からアクセスできる | 必須 |
| F-7 | 入力ソース（ディレクトリ / プロジェクト）を設定で切り替えられる | 必須 |
| F-8 | スマホ縦画面でも読めるレスポンシブ表示 | 重要 |
| F-9 | バリデーションエラー時に、画面上にエラー箇所を表示（落とさない） | 重要 |
| F-10 | `ref` ノードを対象ブロックへのリンクとして表示 | 任意 |
| F-11 | ブロック一覧 / 目次（`sourcePath` 単位）でのナビゲーション | 任意 |
| F-12 | `todo` ノードを視覚的に強調表示 | 任意 |

## 6. 非機能要件

- **更新レイテンシ**: 保存からブラウザ反映まで概ね **2 秒以内**。
- **起動の手軽さ**: コマンド 1 つで起動できる。
- **依存の軽さ**: 重厚なフレームワークは避ける。KaTeX とライブリロード機構程度に留める。
- **オフライン動作**: LAN 内で完結。外部 CDN に依存しない（KaTeX はローカル同梱が望ましい）。
- **疎結合**: 入力ソース（各プロジェクトの構造化テキスト）を**改変せずそのまま読む**。
  アプリ側に研究ドメイン固有のロジックを持ち込まない。

## 7. 接続・セキュリティ前提

- 想定は**信頼できる同一 LAN**（自宅 / オフィス）内での閲覧。
- 当面、認証なし・read-only で許容する。
- 外出先からの閲覧（NAT 越え）が必要になった場合は、トンネリング
  （例: `cloudflared` / `tailscale` / `ngrok` 等）で URL を露出する案を別途検討。
  → これは In scope 外の将来拡張とする。

## 8. 技術方針

確定済み。設計判断とテンプレート（`docs/_template/`）各原則へのマッピング・逸脱根拠は
[architecture.md](./architecture.md) を参照。要点:

- TypeScript（strict）/ pnpm workspace / Biome。
- domain-model（Zod schema + api-contract）を SSOT とする（DB/CRUD 生成は対象外）。
- backend: Fastify + Clean Architecture（gateway のみ、repository 無し）。
- frontend: React + Vite + Tailwind + TanStack Query、FSD 単一ページ。
- 数式: KaTeX バンドル（CDN 不使用）。ライブリロード: SSE。
- backend が frontend ビルド成果物と API/SSE を同一ポート（既定 4321, `0.0.0.0`）で配信。

## 9. 受け入れ基準（Acceptance Criteria）

- [ ] ローカルで 1 コマンドでプレビューサーバが起動する。
- [ ] ローカルのブラウザで入力ソースの全ブロックが KaTeX 付きで表示される。
- [ ] 同一 LAN のスマホからローカル機の URL を開いて同じ表示が見られる。
- [ ] 入力ソースを編集して保存すると、スマホ側が手動操作なしで 2 秒以内に更新される。
- [ ] バリデーションエラーがあっても画面が落ちず、エラー内容が表示される。
- [ ] 入力ソースを設定で差し替えられる（特定プロジェクトに固定されない）。

## 10. 決定事項（旧・未決事項）

- 入力ソースの指定方法 → **環境変数 + CLI 引数**（`config.ts`）。既定はリファレンス入力。
- 表示順序 → ファイル名（`000_`, `001_` …）の連番順で全結合表示。
- ナビゲーション（F-11）→ 初版は連番順の単一ページ。目次/個別ページは任意で後続。
- KaTeX の同梱 → **npm 依存をバンドル**（CDN 不使用、オフライン動作）。
- ポート → 既定 4321（`0.0.0.0` バインド）。固定 IP 運用は利用者の環境設定に委ねる。
- 外出先からの閲覧（§7 トンネリング）→ 当面 out of scope。
