# 構造化テキストのリアルタイム Web プレビュー — 要件定義

## 1. 目的（やりたいこと）

Mac ローカルで編集している「tex 入り構造化テキスト」（`structured-latex/`）の
**レンダー結果**を、**別 PC やスマホのブラウザからリアルタイムに閲覧**できるようにする。

- 編集は引き続き Mac ローカルで行う（エディタは問わない）。
- 編集内容が保存されると、閲覧側のブラウザ表示が**自動で更新**される（手動リロード不要）。
- 数式は KaTeX でレンダリングする（データモデルが既に KaTeX 前提）。

## 2. 用語と前提

- **構造化テキスト**: `structured-latex/content/*.mjs` が export する `ConvertedBlock[]`。
  スキーマは [`../schema.d.ts`](../schema.d.ts) を参照。
  - 数式は LaTeX 文字列（`math` / `displayMath` ノード）として保持されている。
  - 各ブロックは `kind`（theorem/definition/claim/remark/note）, `title`, `statement`,
    `proof`, `notes`, `sourcePath`, `sourceOrdinal` 等を持つ。
- **レンダー結果**: 上記ブロック列を HTML 化し、数式を KaTeX で描画したもの。
- 検証は `node structured-latex/tools/validate-content.mjs`（現在 123 blocks 通過）。

## 3. スコープ

### 3.1 In scope（今回やる）
- ローカルの構造化テキストを HTML + KaTeX で表示する**ビューア**。
- ファイル変更を検知して閲覧ブラウザを**自動更新**する**ライブリロード**。
- **同一ネットワーク内**の別 PC / スマホからの**閲覧**（read-only）。

### 3.2 Out of scope（今回やらない）
- ブラウザ側からの**編集**・双方向同期（閲覧専用でよい）。
- 複数人での同時編集・コンフリクト解決。
- 認証・アクセス制御の作り込み（§7 でゆるい前提を置く）。
- 構造化テキスト自体の内容変換（既存の変換ツールの責務）。
- Typst / PDF 出力の置き換え（別軸の作業）。

## 4. 利用シナリオ

1. Mac で `structured-latex/content/*.mjs` を編集して保存する。
2. Mac 上でプレビュー用プロセス（ローカルサーバ）を 1 つ起動しておく。
3. 別 PC やスマホのブラウザで Mac の URL（例: `http://<MacのLAN IP>:<port>/`）を開く。
4. Mac 側で保存するたびに、別 PC / スマホの表示が**数秒以内に自動更新**される。

## 5. 機能要件

| ID | 要件 | 優先度 |
|----|------|--------|
| F-1 | `content/*.mjs` を読み込み、全ブロックを HTML として表示する | 必須 |
| F-2 | `math` / `displayMath` を KaTeX で描画する | 必須 |
| F-3 | `kind` ごとに見出し・体裁を分けて表示（theorem/definition/claim/remark/note） | 必須 |
| F-4 | `title`, `statement`, `proof`, `notes` を構造的に表示する | 必須 |
| F-5 | ファイル保存を検知し、閲覧ブラウザを自動更新する（ライブリロード） | 必須 |
| F-6 | LAN 内の別端末（PC / スマホ）からアクセスできる | 必須 |
| F-7 | スマホ縦画面でも読めるレスポンシブ表示 | 重要 |
| F-8 | バリデーションエラー時に、画面上にエラー箇所を表示（落とさない） | 重要 |
| F-9 | `ref` ノードを対象ブロックへのリンクとして表示 | 任意 |
| F-10 | ブロック一覧 / 目次（`sourcePath` 単位）でのナビゲーション | 任意 |
| F-11 | `todo` ノードを視覚的に強調表示 | 任意 |

## 6. 非機能要件

- **更新レイテンシ**: 保存からブラウザ反映まで概ね **2 秒以内**。
- **起動の手軽さ**: コマンド 1 つで起動できる（`node ...` 1 行程度）。
- **依存の軽さ**: 重厚なフレームワークは避ける。KaTeX とライブリロード機構程度に留める。
- **オフライン動作**: LAN 内で完結。外部 CDN に依存しない（KaTeX はローカル同梱が望ましい）。
- **既存資産の不変更**: `schema.mjs` / `content/*.mjs` を改変せずそのまま読む。

## 7. 接続・セキュリティ前提

- 想定は**信頼できる同一 LAN**（自宅 / オフィス）内での閲覧。
- 当面、認証なし・read-only で許容する。
- 外出先からの閲覧（NAT 越え）が必要になった場合は、トンネリング
  （例: `cloudflared` / `tailscale` / `ngrok` 等）で URL を露出する案を別途検討。
  → これは In scope 外の将来拡張とする。

## 8. 技術方針（案・確定ではない）

最小構成の候補。実装時に確定する。

- **サーバ**: Node の軽量 HTTP サーバ（標準 `http` か最小限の依存）。
  `content/*.mjs` を動的 import し、JSON 化してクライアントへ渡す。
- **レンダリング**: クライアント側 JS でブロック列を DOM 化し、KaTeX で数式描画。
  （または将来的にサーバ側プリレンダ + ハイドレーションも可）
- **ライブリロード**:
  - 案 A: `content/` を `fs.watch` で監視し、変更時に **WebSocket** / **SSE** で
    クライアントへ通知 → クライアントが再フェッチ & 再描画。
  - 案 B: ブラウザが一定間隔でポーリングし、データのハッシュ差分で再描画。
  - → リアルタイム性と単純さのバランスから **SSE（案 A）** を第一候補とする。
- **配信範囲**: サーバを `0.0.0.0` にバインドして LAN 公開。port は固定（例: 4321）。

## 9. 受け入れ基準（Acceptance Criteria）

- [ ] Mac で 1 コマンドでプレビューサーバが起動する。
- [ ] Mac のブラウザで全 123 ブロックが KaTeX 付きで表示される。
- [ ] 同一 LAN のスマホから Mac の URL を開いて同じ表示が見られる。
- [ ] `content/*.mjs` を編集して保存すると、スマホ側が手動操作なしで 2 秒以内に更新される。
- [ ] バリデーションエラーがあっても画面が落ちず、エラー内容が表示される。

## 10. 未決事項（実装前に確認したいこと）

- 表示順序: ファイル名（`000_`, `001_` …）の連番順で全結合表示でよいか。
- 目次 / ブロック単体ページのような**ナビゲーション**は初版で必要か（F-10）。
- KaTeX の同梱方法（npm 依存にするか、ベンダリングするか）。
- ポート番号・固定 IP 運用の取り決め（Mac の LAN IP は DHCP で変わり得る）。
- 将来的に外出先からの閲覧（§7 トンネリング）まで見据えるか。
